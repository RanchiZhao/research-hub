# GDPO: Group reward-Decoupled Normalization Policy Optimization

## Problem: Advantage Collapse in Multi-Reward GRPO
- GRPO: sum rewards first, then normalize
- Distinct reward combinations collapse to identical advantages
- Example: 2 binary rewards, 6 possible combos → only 2 unique advantage groups
- Removing std normalization alone causes training instability

## Method: GDPO
- **Step 1**: Normalize each reward independently (A_k = (r_k - mean) / std)
- **Step 2**: Sum normalized advantages (A_sum = A_1 + ... + A_N)
- **Step 3**: Batch-wise normalization for stability
- Preserves reward-specific information

## Conditioned Rewards
- Gate easier rewards on harder ones
- R_easy = {R_easy if R_hard ≥ threshold; 0 otherwise}
- Example: length reward only counts when answer is correct
- More controllable than weight tuning

## Experiments
- Tool calling: BFCL-v3, +2.6% (1.5B)
- Math: AIME +6.3% (1.5B), +4.4% (7B)
- Coding: CodeContests +2.4% (7B)
- Length violations cut by 40-58%

## Key Takeaways
- Drop-in replacement for GRPO in multi-reward settings
- Normalize each reward separately before summing
- Use conditioned rewards for different difficulty objectives
